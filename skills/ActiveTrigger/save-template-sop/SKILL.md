# Save Template SOP（流程記錄與摘要）

## 目標

1. **記錄**：依本次對話與（必要時）程式庫查證，整理「使用者問了什麼 → AI 與程式實際做了什麼 → 結論與待辦」。
2. **說清楚頁面**：從入口到畫面，說明**誰呼叫誰**、**資料從哪來**、**模板／元件如何嵌套**。
3. **說清楚 API**：從**對外 URL／HTTP 方法**到**路由／Controller／Service／Model／DB 或外部服務**，再到**回應格式與錯誤路徑**。

## 何時啟用

- **觸發詞 `savesop`**：使用者訊息在 **trim 後、不分大小寫** 等於 `savesop` 時（單獨一行或整則訊息僅此一字），**立即**依本文件「作業順序」執行：整理**截至目前對話**的流程摘要，並寫入 `md文件檔/local_notes/`（勿只回覆「好的」而不產檔）。
- 使用者要「把這次討論的流程記下來」「做重點摘要」「之後能照著懂整條鏈」。
- 使用者要「看懂這頁怎麼 render」「模板從哪 include」「SPA 路由對到哪個元件」。
- 使用者要「這支 API 完整流程」「誰打誰、參數與回傳是什麼」。

## 作業順序（對代理）

1. **對齊範圍**：用 1～2 句重述「本次要摘要的主題」（例如：某頁登入後列表、某支 REST API）。
2. **收斂對話**：從對話中抽出已確認的事實；不確處標成「待查」並到 repo 用搜尋／讀檔補齊，避免只寫推測。
3. **畫兩條鏈**（依專案型態取捨，沒有的區塊標 N/A）：
   - **渲染鏈**：入口（路由／`index.php`／`main.tsx`／`app.vue` 等）→ 版面／父模板 → 子模板或元件 → 資料綁定來源（props／store／server data）。
   - **API 鏈**：Client 呼叫點（`fetch`／axios／表單 action／server action）→ 路由規則 → Handler／Controller → 驗證與授權 → 業務層 → 資料層 → 回應與 HTTP 狀態碼。
4. **附可點據**：每個關鍵步驟盡量給**檔案路徑＋符號**（函式／類別／路由名）；若工作區規則要求可點行號，則附上精簡程式碼引用。
5. **輸出與落檔**：使用下方「輸出範本」產出完整 Markdown 後，**寫入專案根目錄** `md文件檔/local_notes/` 內的新檔案。若該目錄不存在，**先建立** `md文件檔/local_notes` 再寫入。檔名建議：`YYYY-MM-DD-<簡短主題-kebab>.md`；若使用者指定檔名或子路徑，以使用者為準（仍應落在 `local_notes` 下，除非使用者明確要求其他位置）。完成後在對話中回報**實際儲存路徑**。

## 落檔位置（本專案約定）

- **根路徑**（相對於 repo 根目錄）：`md文件檔/local_notes/`
- **用途**：本 skill 產生的流程摘要、渲染鏈與 API 鏈筆記。
- **編碼**：UTF-8。

## 渲染鏈必查項目（檢查清單）

- 使用者如何進入此畫面（URL、路由名、選單或 redirect）？
- 主框架／layout 與內容區誰包誰？是否有 **partial／component／slot／yield**？
- 資料是 **SSR、CSR 初次載入、還是客戶端再請求**？資料在何時注入 view？
- 有無 **快取、權限 gate、條件分支** 會改變渲染結果？

## API 鏈必查項目（檢查清單）

- **Method + Path**（或 RPC 名稱）與是否需 **認證／CSRF／CORS**。
- **Request**：query／body／header 關鍵欄位；**Response**：成功與常見錯誤的形狀與狀態碼。
- 與資料庫或外部 API 的邊界在哪一層？**交易／重試／幂等** 若對理解有幫助則一句帶過。

## 注意

- 專案若為 **Yii／Laravel／Rails／Next.js** 等，用該框架慣用詞（action、view、partial、blade、controller）稱呼即可，不必硬套其他框架名稱。
